IMAGEtoCAD

Convert technical blueprint images into production-ready 2D DXF files for CNC machining.

IMAGEtoCAD is a self-contained Python application that takes uploaded technical blueprint images (containing multiple 2D orthographic views with dimensions), processes them through a computer vision pipeline, extracts geometry and dimensional annotations, and exports clean, layered DXF files suitable for CAM toolpath generation.

Features

  • Multi-view segmentation — Automatically separates Top, Front, and End views from a single blueprint image using whitespace gap analysis and contour bounding boxes
  • Line detection & merging — Probabilistic Hough transform detects line segments; graph-based clustering with PCA principal axis projection merges broken fragments into continuous segments
  • Arc fitting — Least-squares circle fitting (Taubin's algebraic method + Gauss-Newton refinement) converts pixelated corner contours into true CAD arc entities with proper winding order
  • OCR dimension extraction — EasyOCR (with Tesseract fallback) extracts dimensional text and associates it with geometry using multi-factor scoring
  • Scale calibration — Auto-detects pixel-to-unit ratio from dimension text; manual override available when OCR fails
  • Clean DXF export — ezdxf-based generation with CAM-standard numeric-prefixed layers
  • High-DPI print preview — Matplotlib-based rasterization at configurable DPI (300/450/600)
  • Web-based UI — FastAPI + Jinja2 single-page application with drag-drop upload
  • No jagged polylines — Radiused corners become true DXF ARC entities, not approximated line segments
  • CAM-ready layers — Cutting layers (01–09) separated from non-cutting reference layers (90–99)
  • Coordinate transformation — Pixel space (top-left origin, y-down) → engineering space (bottom-left origin, y-up) with scale division
  • Defensive validation — Magic byte image verification, zero-division guards, bounds-clamped ROI extraction, RGBA channel handling

Pipeline Flow

Step-by-step processing from uploaded image to exported DXF.

User drops a blueprint image (PNG/JPG/BMP/TIFF) into the web UI. The backend validates the file using magic byte signatures (PNG header, JFIF, BMP, TIFF) before processing. Corrupted or non-image files are rejected immediately.

The image undergoes a standard computer vision preprocessing chain:

  1. Grayscale conversion — RGB/BGRA → single channel
  2. Gaussian denoise — 5×5 kernel blur to reduce sensor noise
  3. Adaptive threshold — Gaussian-weighted local threshold (block size 15, C=8) produces clean binary image
  4. Morphological close — 2×2 kernel fills small gaps in drawn lines
FeatureMethodOutput
LinesHoughLinesP (probabilistic)Line segments with endpoints
CirclesHoughCircles (gradient)Center + radius
CrosshairsContour aspect ratio analysisInferred bore centers
ContoursRETR_EXTERNAL + RETR_LISTRaw point sequences for arc fitting

View regions are separated using two complementary methods:

  1. Contour bounding boxes — Large external contours correspond to individual view frames
  2. Projection profiles — Horizontal/vertical pixel density histograms find whitespace gaps between views

Views are labeled automatically (TOP VIEW, FRONT VIEW, END VIEW) based on position.

EasyOCR extracts text from each view region. Text-to-geometry association uses multi-factor scoring:

  • Perpendicular distance from text center to line
  • Angular alignment (dimension text is typically parallel to the feature)
  • Endpoint proximity (text near extension line terminations)

The pixel-to-unit ratio is computed by:

  1. Finding an OCR dimension value (e.g., "7.00"")
  2. Associating it with the nearest detected line
  3. Computing: scale = line_pixel_length / dimension_value

If auto-calibration fails, the user can enter a manual scale override in the UI.

The GeometryOptimizer.merge_collinear_lines() method:

  1. Builds a connectivity graph using union-find (angle + distance criteria)
  2. Finds connected components
  3. Computes principal axis via PCA for each component
  4. Projects all endpoints onto the axis
  5. Creates merged line from min to max projection

Two arc detection methods work in parallel:

  • Contour-based — Extracts contours, filters by length, fits circles via Taubin method
  • Profile corner-based — Computes turning angles on the outer contour, identifies corner regions, fits circles

Winding order is checked via the signed area (shoelace formula) to ensure DXF arcs are drawn counter-clockwise.

The CADGenerator writes all geometry to an ezdxf document with proper layering, scale-calibrated coordinates, and Y-axis flip. Output format is DXF R2018 for broad CAD compatibility.

The PreviewEngine renders the DXF to a high-DPI PNG/JPEG using matplotlib. Text sizing is computed relative to drawing bounds for readability at any scale. The DXF is saved to the user-specified folder.

Layer Convention

DXF output uses numeric-prefixed layer names for CAM software compatibility.

LayerColorPurposeCAM Usage
01_CUT_OUTLINEWhite (7)Outer profile, primary cutting geometryToolpath: profile cut
02_CUT_INNERWhite (7)Internal cutouts and pocketsToolpath: pocket cut
03_DRILL_BORESCyan (4)Through holes, counterbores, circlesToolpath: drill cycle
04_MILL_SLOTSGreen (3)Elongated holes, milled featuresToolpath: slot mill
90_CENTER_MARKSRed (1)Crosshair center marksReference only
95_DIMENSIONSYellow (2)Dimension text annotationsReference only
98_TEXTYellow (2)General text and notesReference only
99_REFERENCEGray (8)View boundaries, construction linesReference only

Layers prefixed 01–09 are cutting layers (CAM toolpaths). Layers 90–99 are non-cutting reference layers.

Use CADGenerator.is_cutting_layer(name) to programmatically filter cutting layers from reference layers.

Professional CAM software (VCarve, SheetCAM, Fusion 360) reads these sorted layers sequentially, allowing a machinist to instantly bind tooling paths to specific layers without manual sorting.

Configuration

Tunable parameters for each pipeline module.

vision_engine = VisionEngine(
    line_merge_distance=8.0,      # Max pixel gap to merge lines
    line_merge_angle_tol=10.0,    # Max angle diff (degrees) to merge
    use_easyocr=True,             # True = EasyOCR, False = Tesseract
)
preview_engine = PreviewEngine(
    target_dpi=300,               # Output resolution
    min_width_px=3000,            # Minimum output width
    min_height_px=2000,           # Minimum output height
    padding_frac=0.08,            # Padding around drawing bounds
    background_color="#ffffff",   # Background color
)
optimizer = GeometryOptimizer(
    line_angle_tolerance_deg=5.0,  # Tighter = more selective merging
    line_distance_tolerance=4.0,   # Larger = more aggressive merging
    arc_max_residual=1.5,          # Lower = stricter arc fit quality
    arc_min_points=6,              # Higher = fewer false arc detections
)

Troubleshooting

Common issues and their solutions.

  • Ensure the blueprint has clear, high-contrast lines on a light background
  • Try increasing image resolution or adjusting contrast before upload
  • Check that min_line_length in detect_lines isn't too high for your image
  • The OCR engine couldn't find a dimension value near a detected line
  • Use the Manual Scale Override field in the UI: measure a known dimension in pixels, divide by the real value
  • Example: if a 7.50" dimension spans 150 pixels, enter 20.0 (150/7.5) with unit "in"
  • Check the layer visibility in your CAD software — reference layers (90–99) may be hidden
  • Verify the scale factor in the export response matches your expected units
  • Open the DXF in a viewer that supports R2018 format (AutoCAD 2018+, LibreCAD, QCAD)
  • The DXF may have no entities — check the entity counts in the save response
  • Try a higher DPI setting (450 or 600) for better text visibility
  • Ensure the blueprint image has detectable geometry (lines, circles, contours)
  • EasyOCR first-run downloads language models (~40MB) — this is normal
  • Subsequent runs are faster as models are cached
  • All CPU-bound operations run via run_in_threadpool to avoid blocking the event loop
  • For production, consider running with a GPU-enabled EasyOCR build

Known Limitations

  • OCR accuracy — Small, skewed, or overlapping dimension text may not be extracted correctly
  • View segmentation — Extremely dense blueprints with minimal whitespace between views may not segment cleanly
  • Arc detection range — Contour-based arc detection filters by length (20–500 px). Very small fillets or very large arcs may be missed
  • Memory usage — EasyOCR model loading requires ~2GB RAM. HD preview at 600 DPI may require additional memory
  • Single-session — Sessions expire after 1 hour. No persistent storage between server restarts
