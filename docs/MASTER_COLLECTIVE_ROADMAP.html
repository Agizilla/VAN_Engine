📐 APPENDIX B — Coding Standards

CODING_STANDARDS.md — Click to expand

Python

  • Use type hints for all function signatures (PEP 484).
  • Prefer dataclasses over plain dicts for structured data.
  • Use with for resource management.
  • PEP 8 naming: snake_case for variables/functions, PascalCase for classes.
  • Use logging.getLogger(__name__) — never print().
  • For async I/O, prefer asyncio with async/await.
  • All audio processing must be idempotent and deterministic for repeatable results across the mesh.

C#

  • .NET naming: PascalCase public, camelCase params/locals.
  • Use async/await for I/O — never .Result or .Wait() in production.
  • Prefer DI over static singletons.
  • Use record types for immutable DTOs.
  • Use using or await using for disposables.
  • Log with ILogger<T> — never Console.WriteLine.

JavaScript

  • Avoid jQuery for new features — use native DOM APIs.
  • Use ES6+ syntax: const/let, arrow functions, template literals.
  • Prefer TypeScript for non-trivial frontend.
  • Use fetch with response.ok error handling.
  • Never inline event handlers in HTML — use addEventListener.

Security (Cross-Language)

  • Never log secrets (passwords, API keys, tokens, PII). Sanitize before logging.
  • Validate all external input via schema validator (pydantic/zod/JsonSerializer).
  • Use parameterised queries or ORM — never concatenate input into SQL.
  • Store secrets in env vars or secrets manager — never in source control.
  • Implement rate limiting on public endpoints.

Performance

  • Avoid sync blocking calls in async contexts.
  • Use connection pooling for databases and HTTP clients.
  • Implement retries with exponential backoff for transient failures.
  • Use caching for expensive reads that tolerate staleness.
  • Use asyncio.gather over sequential await for independent I/O.

Project Structure

  • Separate concerns: UI, business logic, data access, config in distinct modules.
  • Use dependency inversion (program to interfaces, not implementations).
  • Keep config out of code — use env vars, config files, or config service.
  • Pin dependencies to exact versions (commit lock files).

Git Conventions

  • Use Conventional Commits: type(scope): subject
  • Types: feat, fix, docs, style, refactor, perf, test, chore
  • Keep commits atomic (one logical change per commit).
  • Write commit messages in imperative present tense ("Add validation").
  • Reference issue numbers in commit body when applicable.

Documentation

  • Every public API must have a docstring describing purpose, params, return, exceptions.
  • Use auto-generated API docs (Swagger/OpenAPI for web, Sphinx for Python, DocFX for C#).
  • Inline comments only for why, not what.
