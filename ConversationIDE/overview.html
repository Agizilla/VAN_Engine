Conversation-IDE

A governed local workspace for VAN_Engine.

Conversation-IDE is designed as an operator cockpit for chat, files, builds, audits, and VAN_Engine inspection. It aims to feel like a real control room: local-first, project-aware, and governed by ISO rules rather than a plain prompt box.

Core promise
A single local workspace that can inspect, explain, patch, build, and verify.
Behavior
Governed actions with file access control, audit logging, and rule-aware operations.
Integration
Electron shell, preload bridge, renderer UI, Python/VAN engine bridge, and build tooling.

Key Highlights

These are the elements that make the application feel distinct rather than like a generic IDE wrapper.

  • Governed actions: ISO rules can block, warn, and audit changes instead of only being visible in a panel.
  • Project-aware workspace: the app is intended to know the current repo, file tree, and operational state.
  • Operator mode: the chat surface is meant for inspect, patch, build, and verify workflows.
  • Local-first posture: the tool is designed to keep the core loop offline and self-contained.
  • Audit visibility: state changes should be traceable through an explicit log and status surfaces.

What Users Get

Conversation-IDE is positioned as a workspace where the assistant can act like an operator, not just a text generator.

  • Chat with project context and direct action hooks.
  • Browse, edit, and protect local files in a governed way.
  • Run builds and surface progress in the same environment.
  • Inspect VAN_Engine state, rules, and audit signals without leaving the app.

Visual Direction

The interface should feel deliberate and technical, with a dark control-panel aesthetic rather than a generic chat client.

  • Deep navy backgrounds with cyan and blue accents.
  • Layered glass panels, thin borders, and compact spacing.
  • Terminal-like editor surfaces and operational status chips.
  • Clear hierarchy between chat, file tree, build, and governance panels.

This page is a concept overview and visual direction reference. The real implementation should keep the same governed, local-first intent while aligning the UI and IPC contract in `OpenCode.md`.
