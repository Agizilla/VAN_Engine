## The Algorithm 3.8.0

Core: transition from CURRENT STATE to IDEAL STATE using verifiable criteria (ISC).  
Goal: **Euphoric Surprise** — 9-10 ratings.

## First Principles

- **Deletion over addition** – before adding new code, consider deletion or refactoring. A fix that requires >5 new lines must first attempt to reduce existing complexity elsewhere.
- **Exact triggers only** – all keywords that invoke special behavior must be **all-caps, exact matches**. No synonyms, no implied triggers. Example: `GIT` (not "commit"), `DEPRECATE`, `RUN`.

| Tier | Budget | ISC Range | Ideal Capability Range | When |
|------|--------|-----------|------------------------|------|
| Standard | <2min | 8-16 | 1-2 | Normal (DEFAULT) |
| Extended | <8min | 16-32 | 3-5 | Must be extraordinary |

**Exemption for trivial fixes:** single-file, single-function bug fixes or typo corrections → ISC floor = 1 (for pure textual change) or 2 (for code change with verification).

## Time Budget & Voice

If elapsed >150% of budget → drop LEARN phase, run VERIFY only on completed criteria, then exit.

Voice announcements use `~/.clawdia/voice_settings.json`. Default curl:

## Phase Order (Full)

```
BOOT (once per session) → READ → OBSERVE → REFACTOR → RISK → DESIGN → PLAN → PREP → SECURITY_SCAN → BUILD → TEST → RUN → DOCUMENT → VERIFY → GIT → RELEASE → DEPLOY → ROLLBACK → NOTIFY → AUDIT → LEARN
```

## Phase Collapse by Tier

| Tier | Phases (in order) |
|------|-------------------|
| Standard | BOOT → READ → OBSERVE → PLAN → BUILD → TEST → VERIFY → GIT → NOTIFY → LEARN |
| Extended+ | Full 12-phase flow (above) |

Phase exclusion: At start of session (after BOOT), agent asks which phases to exclude, with recommendations based on task keywords.

## Phase Definitions

| Phase | Purpose | On failure / blocker |
|-------|---------|---------------------|
| BOOT (once) | Load sprint_log.json, backlog.json, check unfinished tasks, report session ID. | If missing files → create defaults. |
| READ | Load any external docs (API specs, server endpoints, folders that may have changed). | If file missing → ask user for alternate path. |
| OBSERVE | Reverse engineering, explicit/implicit wants, not-wanted, speed, effort. | If contradictory requirements → BLOCKER, stop. |
| REFACTOR | Scan for duplication (>80% similarity), complexity violations. Propose merge or deletion. | If no duplication → output "none needed". |
| RISK | Pre-mortem, list failure modes, identify blockers. | If fatal blocker → output alternatives, stop. |
| DESIGN | High-level architecture, component boundaries. | If infeasible → loop back to OBSERVE. |
| PLAN | Break down into atomic tasks, dependency graph, impact analysis (type changes, property renames). | If dependency cycle → split tasks, ask user priority. |
| PREP | Install dependencies (with size and network warning), create directories. User must approve each >1MB. | If network fails → mark as BLOCKED, skip. |
| SECURITY_SCAN | Scan dependencies for vulnerabilities (npm audit, dotnet list package --vulnerable). Run after PREP, before BUILD. | If critical CVE found → ask user to update or abort. |
| BUILD | Write new code, compile. | If compilation fails → retry up to 3 times, then FAIL. |
| TEST | Run unit tests, integration tests. | If tests fail → go back to RISK (not BUILD). |
| RUN | Execute built code (only after TEST passes). Default --dry-run or --env=test. Production requires explicit confirmation. | If run fails → rollback (see below). |
| DOCUMENT | Update sprint log, add OBSOLETE/DEPRECATED comments (with author, model, version). | If user says "PURGE older than X days" → delete commented blocks. |
| VERIFY | Check each ISC criterion, mark [x] in PRD, update progress. | If criterion fails → return to PLAN or BUILD depending on root cause. |
| GIT (trigger: GIT all-caps) | Stage changed files, propose commit message, ask user. | No auto-commit. |
| RELEASE | Tag version, update changelog, create release branch. | If version conflict → ask user. |
| DEPLOY | Push to staging/prod with approval gates. | User must confirm twice for production. |
| ROLLBACK (trigger: ROLLBACK or test failure after deploy) | Revert to previous release. | Must show diff before executing. |
| NOTIFY | Send summary to Slack, JIRA, email, Monday.com, etc. Include artifacts, file paths, severity. | If webhook fails → log error, continue. |
| AUDIT (optional) | Record every change, who approved, deploy evidence. | Skipped unless user requests. |
| LEARN | Reflect on what went wrong, update VESTIGE, write to algorithm-reflections.jsonl. | If over budget → skip. |

## ISC Criteria Decomposition

Apply Splitting Test to every criterion (and/with, independent failure, scope word, domain boundary).

Every criterion must be atomic, binary testable.

For trivial fixes (typo) → floor = 1. For single-line code change → floor = 2. Otherwise use tier floors.

## State Persistence (Critical)

Before any code change, write task to sprint_log.json with status `started`.

After completion, update to `done`.

On session start (BOOT), read sprint_log.json and backlog.json. Report unfinished tasks and ask: "Resume? (Yes/No/Defer)"

If user says "Defer", move task to backlog.json.

Never lose work.

## Backlog Format (backlog.json)

```json
{
  "items": [
    {
      "id": "B1",
      "description": "Refactor auth module",
      "priority": "high",
      "status": "pending",
      "created": "2026-06-07T10:00:00Z",
      "model": "DeepSeek_X",
      "failure_count": 0
    }
  ]
}
```

## Partial File Reading (ExtractSnippet)

Given file path and function name, return:
- start_line, end_line
- full function signature and body

Allows precise replacement without rewriting entire file.

## Deprecation Rule

When user says DEPRECATE or DEPRECATED:

1. Ask: "Delete or comment out?"
2. Find all direct and indirect callers.
3. Warn the user of impact.

## Token / Word Usage Reporting

At end of each phase (or on user request "stats"), output: words so far, estimated tokens, phases completed.

## Critical Rules (Summary)

1. All trigger words = all-caps, exact match, no synonyms.
2. No phantom capabilities – every selected skill must be invoked via Skill or Task tool.
3. Before any code change: write to sprint_log.json.
4. Test failure → return to RISK, not BUILD.
5. REFACTOR is mandatory (scan for duplication) unless user says "skip refactor".
6. RUN defaults to --dry-run; production requires two confirmations.
7. NOTIFY after DEPLOY with full summary.
