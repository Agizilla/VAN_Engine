## The Algorithm 3.8.1

Core: transition from CURRENT STATE to IDEAL STATE using verifiable criteria (ISC). 
Goal: **Euphoric Surprise** — 9-10 ratings.

### First Principles

*   **Deletion over addition** – before adding new code, consider deletion or refactoring. A fix that requires >5 new lines must first attempt to reduce existing complexity elsewhere.
*   **Exact triggers only** – all keywords that invoke special behavior must be **all-caps, exact matches**. No synonyms, no implied triggers (e.g., `GIT` not "commit").

### Effort Levels & ISC Range

| Tier | Budget | ISC Range | Ideal Capability Range |
|------|--------|-----------|------------------------|
| Standard | <2min | 8-16 | 1-2 |
| Extended | <8min | 16-32 | 3-5 |
| Advanced | <16min | 24-48 | 4-7 |
| Deep | <32min | 40-80 | 6-10 |
| Comprehensive | <120min | 64-150 | 8-15 |

*Exemption for trivial fixes: single-character typo/comment → ISC floor = 1; single-line code change → ISC floor = 2.*

### Time Budget, Voice, & Stats

*   If elapsed >150% of budget → drop LEARN phase, run VERIFY only on completed criteria, then exit.
*   Voice settings: `~/.clawdia/voice_settings.json`. User can say "VOICE Settings" to configure.
*   Stats: Output word count (words), estimated tokens (words x 1.33), and phases completed at the end of every phase.

### Phase Order (19-Phase Flow)

`BOOT → READ → OBSERVE → REFACTOR → RISK → DESIGN → PLAN → PREP → SECURITY_SCAN → BUILD → TEST → RUN → DOCUMENT → VERIFY → GIT → RELEASE → DEPLOY → ROLLBACK → NOTIFY → AUDIT → LEARN`

*   **Standard Tier Collapse**: `BOOT → READ → OBSERVE → PLAN → BUILD → TEST → VERIFY → GIT → NOTIFY → LEARN`.
*   **Phase Exclusion**: At `BOOT`, agent asks which phases to exclude based on task keywords.

### Phase Definitions

| Phase | Purpose | On failure / blocker |
|-------|---------|---------------------|
| BOOT | Load sprint_log.json, backlog.json, check unfinished tasks, report session ID. | Create defaults if missing. |
| READ | Load external docs, API specs, endpoints. | Ask user for alternate path. |
| OBSERVE | Reverse engineering, wants/not-wants, effort. | If contradictory → BLOCKER. |
| REFACTOR | Scan for duplication (>80%) and complexity. | If none → "none needed". |
| RISK | Pre-mortem, failure modes, blockers. | If fatal → stop. |
| DESIGN | Architecture, component boundaries. | If infeasible → loop to OBSERVE. |
| PLAN | Atomic tasks, dependency graph, impact analysis. | If cycle → split tasks. |
| PREP | Install deps, create dirs. User must approve >1MB. | If network fails → skip. |
| SECURITY_SCAN | Scan dependencies (npm audit, etc.). | If CVE found → ask user. |
| BUILD | Write/compile code. | Retry 3x, then FAIL. |
| TEST | Unit/integration tests. | If fail → go to RISK. |
| RUN | Execute (dry-run/test env only). | If fail → ROLLBACK. |
| DOCUMENT | Update sprint log, add DEPRECATED comments. | If "PURGE" → delete. |
| VERIFY | Check ISC criteria, update PRD. | If fail → go to PLAN/BUILD. |
| GIT | Stage, propose message. | No auto-commit. |
| RELEASE | Tag, changelog, branch. | Ask user. |
| DEPLOY | Push to staging/prod. | Double confirmation. |
| ROLLBACK | Revert release. | Show diff first. |
| NOTIFY | Send summary (Slack, JIRA, etc.). | Log error if fails. |
| AUDIT | Record change history. | Optional. |
| LEARN | Reflect, write to reflections.jsonl. | Skip if over budget. |

### State Persistence & Tools

*   **Write-ahead logging**: Before ANY code change, write task to `sprint_log.json` (`started`). Update to `done` on completion.
*   **Resume Flow**: On `BOOT`, report unfinished tasks and prompt: "Resume? (Yes/No/Defer)".
*   **ExtractSnippet**: Use to read/replace functions precisely (start_line, end_line) instead of entire file rewrites.
*   **Deprecation Rule**: If `DEPRECATE` is invoked, ask "Delete or comment out?", identify callers, and warn of impact.

### Critical Rules

1.  All trigger words = **ALL-CAPS, EXACT MATCH**.
2.  No phantom capabilities – tools must be invoked.
3.  Test failure → return to `RISK`, not `BUILD`.
4.  `REFACTOR` is mandatory unless "skip refactor" requested.
5.  `RUN` defaults to `--dry-run`; production requires two confirmations.
6.  `NOTIFY` after `DEPLOY` with full summary.
