1. Executive Summary

ChirpChat is a zero-internet, decentralized mesh messaging application that enables peer-to-peer text messaging, group chat, and file sharing without any cloud infrastructure, cellular connectivity, or internet access.

PAIRING Devices discover and authenticate using ultrasonic audio chirps (19-24kHz, inaudible to humans)

MESH Messages route over Wi-Fi Direct / local hotspot mesh, not the internet

SECURITY End-to-end encrypted with forward secrecy, no server to compromise

OFFLINE Functions in disaster zones, remote areas, air-gapped environments, or any scenario without network infrastructure

2. System Scope

What ChirpChat Is

  • A decentralized messaging app that works without internet, cellular towers, or any centralized infrastructure
  • A mesh network where every device acts as both client and router
  • An ultrasonic pairing mechanism for zero-touch device discovery
  • A privacy-first communication tool with end-to-end encryption by default

What ChirpChat Is Not

  • Not a WhatsApp replacement for online use
  • Not a cloud-backed service
  • Not compatible with standard email or SMS
  • Not a peer-to-peer file sharing network (BitTorrent-style)
  • Not designed for real-time voice/video calls (future consideration)

ISO_004 ChirpChat is a separate project from VAN_Engine. No cross-project merging.

ISO_019 No external bridges or cloud dependencies. Fully offline by default.

PLATFORMS Mobile-first (Android/iOS). Desktop companion (Windows/macOS/Linux) for file transfers.

4. Ultrasonic Pairing Protocol

SPEC Version 1.0

BAND 19,000 Hz - 24,000 Hz (inaudible to most humans, above typical hearing range)

4.1 Frequency Allocations

FrequencyPurpose
19,000 - 19,500 HzDiscovery chirp / presence beacon
19,500 - 20,000 HzHandshake control frames
20,000 - 22,000 HzData payload (key exchange material)
22,000 - 24,000 HzError correction / redundancy guard band

4.2 Chirp Frame Structure

Frame format (each chirp = ~100ms duration): +---------+----------+-----------+-----------+------------+ | Preamble| Sync Word| Header | Payload | CRC-32 | | 16 bits | 16 bits | 16 bits | Variable | 32 bits | +---------+----------+-----------+-----------+------------+ Preamble: 0xAAAA (alternating bits for clock recovery) Sync: 0xCB3C (unique word for frame detection) Header: [4 bits: frame type][4 bits: version][8 bits: length] Payload: Frame-type-specific data (see 4.3) CRC-32: IEEE 802.3 CRC over Header + Payload Modulation: Frequency Shift Keying (FSK) with 2 carriers - Mark (1): center frequency + 200 Hz - Space (0): center frequency - 200 Hz Bit rate: ~200 bps (after error correction)

4.3 Frame Types

TypeCodePayloadDirection
DISCOVER0x01DeviceID (8 bytes) + Capabilities (4 bytes)Broadcast
CHALLENGE0x02Nonce (16 bytes) + Ephemeral Public Key (32 bytes)Unicast response
RESPONSE0x03Signed Nonce (64 bytes) + Ephemeral Public Key (32 bytes)Unicast response
CONFIRM0x04Session Fingerprint (4 bytes) + Connection Token (16 bytes)Unicast
ACK0x05Frame sequence number (2 bytes)Unicast
ERROR0xFFError code (1 byte) + Reason (variable)Unicast

4.4 Handshake Sequence

  1. Discover: Device A broadcasts DISCOVER chirp every 3 seconds while in pairing mode. Includes DeviceID and capabilities flag (supports Wi-Fi Direct, WebRTC, file sharing).
  2. Challenge: Device B hears DISCOVER, responds with CHALLENGE containing a random 16-byte nonce and its X25519 ephemeral public key.
  3. Response: Device A signs B's nonce with its Ed25519 identity key, includes its own ephemeral public key, and sends RESPONSE.
  4. Confirm: Device B verifies the signature, derives shared secret via X25519(eph_sk, eph_pk), sends CONFIRM with first 4 bytes of session fingerprint (SHA-256 of shared secret).
  5. Key Established: Both devices now share a session key. Connection token is passed via audio; devices switch to Wi-Fi Direct for data channel.

4.5 Collision Detection

When multiple devices chirp simultaneously, collisions occur. Mitigation strategies:

  • Random backoff: Each device waits a random interval (500-2000ms) between DISCOVER chirps. The interval is seeded by DeviceID to reduce birthday collisions.
  • Carrier sensing: Before chirping, listen on the discovery frequency for ~50ms. If energy detected, skip this cycle.
  • CRC failure retry: If CRC-32 check fails, receiver sends no ACK. Sender retransmits after backoff (exponential: 100ms, 200ms, 400ms, max 3 retries).
  • Collision domain: The discovery channel is shared; once CHALLENGE/RESPONSE phase begins, devices hop to a frequency derived from the DeviceID pair (frequency = 20,000 + hash(id_a XOR id_b) % 2,000).

4.6 Fallback Pairing

If ultrasonic pairing fails after 3 attempts (ambient noise, hardware filtering, hearing impairment):

  • QR code: Device A displays a QR code encoding the same DISCOVER payload + ephemeral key. Device B scans.
  • NFC: Devices tap to exchange handshake material via NDEF record.
  • Manual code: 6-digit numeric code displayed on one device, entered on the other. Derives ephemeral key from code + device secret.

5. Wi-Fi Mesh Layer

The mesh layer connects devices after ultrasonic pairing establishes trust. It manages node discovery, message routing, and connection maintenance.

5.1 Topology: Flooding Mesh

ChirpChat uses a flooding mesh topology where every message received by a node is re-broadcast to all connected peers within a Time-To-Live (TTL) limit. This is the simplest mesh routing strategy and appropriate for the expected scale (dozens of devices, not thousands).

PropertyValue
Topology typeFlooding mesh (epidemic routing)
Max mesh size256 nodes (target), 64 nodes (v1.0)
Default TTL7 hops
Re-broadcast delay50-200ms jitter (random per hop)

5.2 Node Identity

  • Each device generates an Ed25519 keypair on first launch
  • NodeID = base58(public_key[0:8]) (8-byte truncated hash, collision-resistant for mesh scale)
  • A human-readable display name is settable by the user (default: random adjective-noun pair)
  • NodeID is stable across restarts. Display name can change.

5.3 Node Discovery

After ultrasonic pairing, the connection token is used to establish a transport link:

  1. Wi-Fi Direct (P2P): One device becomes the Group Owner (GO), the other connects as client. GO assignment negotiated during pairing (device with more battery becomes GO).
  2. mDNS: Each node announces itself via mDNS on the local subnet with service type _chirpchat._tcp. Nodes previously paired by audio can reconnect via mDNS without re-chirping.
  3. Periodic presence: Every 30 seconds, each node broadcasts a lightweight PRESENCE chirp (ultrasonic) containing only NodeID + IP hash for re-discovery.

5.4 Message Propagation

Message envelope (routing layer): { "msg_id": "base58(32-byte random)", "origin_node": "NodeID of sender", "sender_node": "NodeID of last hop", "ttl": 7, "hop_count": 0, "payload_hash": "SHA-256 of encrypted payload", "payload_size": 1234, "payload": "encrypted message blob (base64)" } Routing algorithm: 1. Receive message 2. If msg_id in seen_cache (last 10,000 messages), DROP 3. If ttl <= 0, DROP 4. Store encrypted blob (for sync) 5. Decrypt if intended recipient or group member 6. Decrement ttl, increment hop_count 7. Forward to all connected peers (except origin_node) 8. Wait random jitter (50-200ms) before each forward

5.5 NAT / Network Boundaries

  • Wi-Fi Direct: Operates at Layer 2, bypasses IP routing entirely. No NAT issues.
  • LAN (same subnet): mDNS + TCP direct. No NAT.
  • Mesh bridge: A node connected to two separate mesh islands (e.g., in two rooms) acts as a bridge. It re-broadcasts messages between the islands. No STUN/TURN needed.
  • Cellular fallback: Explicitly disabled per ISO_019. ChirpChat never uses cellular data unless user enables bridge mode.

5.6 Database Sync / Conflict Resolution

  • Each message has a Lamport timestamp (node clock + NodeID tiebreaker) for causal ordering
  • Conflicts (concurrent edits on different nodes) are resolved by last-writer-wins based on Lamport timestamp
  • Deleted messages are tombstoned (deletion marker + timestamp), never truly erased from mesh
  • Each node syncs its message store incrementally when they detect a peer with a higher max timestamp

6. Message Protocol

The application-layer message protocol defines how messages are structured, serialized, encrypted, and delivered.

6.1 Wire Format: Protocol Buffers

Messages are serialized using Protocol Buffers (protobuf) for compact binary encoding and cross-platform compatibility. Protobuf was chosen over JSON for smaller wire footprint (~30% smaller) which matters for ultrasonic backup channels.

syntax = "proto3"; message Envelope { string msg_id = 1; // Unique message ID (base58) string conversation_id = 2; // Conversation identifier uint64 timestamp = 3; // Lamport timestamp (milliseconds) MessageType type = 4; // Message type enum bytes encrypted_payload = 5; // AEAD-encrypted Content bytes sender_signature = 6; // Ed25519 signature of (msg_id + timestamp + encrypted_payload) uint32 ttl = 7; // Remaining TTL string origin_node = 8; // Originating NodeID } enum MessageType { TEXT = 0; IMAGE = 1; FILE_METADATA = 2; FILE_CHUNK = 3; TYPING = 4; DELIVERY_RECEIPT = 5; READ_RECEIPT = 6; GROUP_CREATE = 7; GROUP_ADD = 8; GROUP_REMOVE = 9; } message Content { oneof content { TextMessage text = 1; FileMetadata file_meta = 2; FileChunk file_chunk = 3; TypingIndicator typing = 4; Receipt receipt = 5; GroupAction group = 6; } } message TextMessage { string body = 1; // UTF-8 text (max 4096 bytes) repeated string mentions = 2; // Mentioned NodeIDs string reply_to = 3; // Optional: msg_id being replied to }

6.3 Delivery Semantics

SemanticUse CaseMechanism
At-least-onceText, images, file metadata, group actionsReceiver sends DELIVERY_RECEIPT. Sender stores msg_id + pending_ack. Retransmits every 5s until ACK received or TTL expires. Duplicates deduplicated by msg_id.
At-most-onceTyping indicators, delivery/read receiptsFire-and-forget. No ACK. No retransmission. Loss is acceptable.
Best-effortFile chunksSent over mesh without ACK per chunk. Receiver requests missing chunks via dedicated REQUEST message. Sender retransmits on request.

6.4 Group Chat Model

  • Groups are broadcast groups: each message is encrypted with a symmetric group key and broadcast to all mesh peers.
  • Only group members can decrypt. Non-members store the encrypted blob and forward it, but cannot read it.
  • Group membership is tracked via an append-only log of GROUP_CREATE, GROUP_ADD, GROUP_REMOVE messages.
  • Group key rotates on member removal (new key distributed to remaining members via pairwise channels).
  • Maximum group size: 32 members (v1.0).

7. File Sharing

File sharing uses erasure coding to survive message loss in the mesh. Instead of reliability at the transport layer, reliability is built into the file encoding.

7.1 Erasure Coding: Reed-Solomon

ParameterValueRationale
AlgorithmReed-Solomon (GF(2^16))Proven, well-optimized, constant-time implementations available
Data chunks (k)8Good balance of overhead vs. recovery
Parity chunks (m)425% overhead; survives loss of up to 33% of chunks
Total chunks (n)12k + m
Chunk size64 KBFits within typical MTU for mesh; avoids fragmentation

Encoding process:

  1. Input file is split into 64 KB blocks. If the last block is smaller, it is zero-padded.
  2. Each block is grouped into sets of 8 (k). The 8 data chunks + 4 parity chunks are computed via Reed-Solomon matrix multiplication.
  3. Each chunk is assigned a unique (block_index, chunk_index) coordinate.
  4. Chunks are broadcast as FILE_CHUNK messages over the mesh.
  5. Receiver collects any 8 of 12 chunks per block to reconstruct. Lost chunks are recoverable from parity.

7.2 File Metadata Schema

message FileMetadata { string file_id = 1; // Unique ID for this transfer string file_name = 2; // Original filename (UTF-8) uint64 file_size = 3; // Size in bytes string mime_type = 4; // MIME type (e.g., "image/jpeg") string sha256_hash = 5; // SHA-256 of original file uint32 total_blocks = 6; // Total number of blocks uint32 chunk_size = 7; // Chunk size in bytes (65536) uint32 k = 8; // Data chunks per block (8) uint32 m = 9; // Parity chunks per block (4) repeated string recipient_ids = 10; // Intended recipients (NodeIDs) }

7.3 Chunk Reassembly

  1. Receiver accumulates FILE_CHUNK messages, indexed by (block_index, chunk_index).
  2. Once any 8-of-12 chunks per block arrive, decode the block via Reed-Solomon and write to disk.
  3. After all blocks are decoded, concatenate in order and verify SHA-256 hash.
  4. If hash matches, emit a delivery receipt with file_id. If not, request missing chunks from sender.
  5. Missing chunk request: unicast a REQUEST_CHUNK message to the sender node containing (file_id, block_index, chunk_index). Sender retransmits on request.

7.4 Thumbnail Previews

  • Images: sender generates a 256x256 JPEG thumbnail (max 10 KB) and sends it inline as the IMAGE message. Full-resolution file is erasure-coded separately.
  • Files: no thumbnail. Display icon based on MIME type.

8. Security Model

DESIGN GOAL Provide end-to-end encryption with forward secrecy, authenticated by ultrasonic proximity. No server can be compromised because no server exists.

8.1 Device Identity

  • Long-term keypair: Ed25519 (signing) generated on first launch. Stored encrypted in device keychain (Android keystore / iOS Secure Enclave).
  • NodeID: base58(truncate(SHA-256(public_key), 8)). Displayed as a human-readable checksum.
  • Display name: User-settable alias. Not trusted for authentication. Verified by NodeID.
  • Identity fingerprint: 12-word BIP39 mnemonic derived from public key for out-of-band verification.

8.2 Key Exchange: X3DH (Extended Triple Diffie-Hellman)

ChirpChat uses the X3DH key agreement protocol (as used in Signal) to establish a session key between two devices:

  1. Each device maintains an identity key (IK), a signed pre-key (SPK), and a one-time pre-key (OPK) bundle.
  2. During ultrasonic pairing, devices exchange IK + ephemeral key via CHALLENGE/RESPONSE frames.
  3. The shared secret is derived as: DH(IK_A, SPK_B) || DH(EK_A, IK_B) || DH(EK_A, SPK_B) || DH(EK_A, OPK_B)
  4. The session key is derived from the shared secret via HKDF. It seeds a double ratchet for forward secrecy.

8.3 Forward Secrecy: Double Ratchet

  • After initial key exchange, messages use the Signal Double Ratchet algorithm for continuous forward secrecy.
  • Each message advances a ratchet step. Compromising the current session key does not reveal past messages.
  • Each message includes a new ephemeral public key (DH ratchet step) so that compromise of one device's keys does not reveal future messages.

8.4 Message Encryption

  • Algorithm: AES-256-GCM (or XChaCha20-Poly1305 on platforms with hardware acceleration)
  • Each message uses a unique nonce (12 bytes), derived from message sequence number + ratchet step
  • Encrypted payload: nonce || ciphertext || auth_tag
  • Messages are signed by sender's Ed25519 key (for non-repudiation within the mesh)

8.5 Trust-On-First-Use (TOFU)

  • The first time two devices pair via ultrasonic, they exchange identity keys in person via audio. This establishes trust by physical proximity.
  • The device stores the peer's identity key permanently. Subsequent reconnections (via mDNS or Wi-Fi Direct) re-use the stored key.
  • If a peer's key changes, the device shows a warning: "Alice's identity has changed. This could mean Alice got a new device OR someone is intercepting your connection. Verify by chirping in person."
  • Users can verify keys out-of-band via the 12-word mnemonic fingerprint or by re-chirping.

8.6 Local Data Encryption

  • All messages are stored encrypted at rest using AES-256-GCM with a key derived from the device's master secret.
  • Master secret is derived from: PKDF2(user_passphrase || device_keychain_secret, salt, 100000 iterations)
  • If no passphrase is set, the master secret is derived from device keychain only (weaker but more convenient).
  • Database file is SQLite with encrypted WAL. Page-level encryption via SQLCipher or similar.

8.7 Security Threat Model

ThreatMitigation
Eavesdropping on ultrasonic pairingPairing requires physical proximity (~10m). Key exchange material is ephemeral and signed. Attacker would need to be in the same room.
Man-in-the-middle during pairingTOFU with physical presence verification. Fingerprint verification via 12-word mnemonic.
Compromised mesh nodeEnd-to-end encryption limits impact. Compromised node can only see messages it has keys for. It cannot read group messages it's not a member of.
Message replayLamport timestamps + message ID deduplication. Replayed messages are dropped.
Denial of service (message flood)Rate limiting per peer (100 msg/s max). TTL bounds prevent infinite floods.
Physical device theftLocal data encrypted at rest with passphrase. Keystore/SE protects keys.

9.1 Local Database Schema

SQLite with encrypted WAL via SQLCipher.

-- Core tables CREATE TABLE conversations ( id TEXT PRIMARY KEY, -- base58(conversation_id) type INTEGER NOT NULL, -- 0=direct, 1=group name TEXT, -- Group name (null for direct) created_at INTEGER NOT NULL, -- Unix timestamp (ms) last_message_at INTEGER, -- Unix timestamp (ms) unread_count INTEGER DEFAULT 0 ); CREATE TABLE messages ( id TEXT PRIMARY KEY, -- msg_id conversation_id TEXT NOT NULL, -- FK to conversations sender_id TEXT NOT NULL, -- NodeID of sender type INTEGER NOT NULL, -- MessageType enum body BLOB, -- Encrypted payload (decrypted on read) timestamp INTEGER NOT NULL, -- Lamport timestamp (ms) status INTEGER DEFAULT 0, -- 0=sending, 1=sent, 2=delivered, 3=read, 4=failed reply_to TEXT, -- msg_id being replied to (nullable) FOREIGN KEY (conversation_id) REFERENCES conversations(id) ); CREATE TABLE file_transfers ( file_id TEXT PRIMARY KEY, conversation_id TEXT NOT NULL, file_name TEXT NOT NULL, file_size INTEGER NOT NULL, mime_type TEXT, sha256_hash TEXT NOT NULL, total_blocks INTEGER NOT NULL, received_blocks INTEGER DEFAULT 0, status INTEGER DEFAULT 0, -- 0=receiving, 1=complete, 2=failed FOREIGN KEY (conversation_id) REFERENCES conversations(id) ); CREATE TABLE group_members ( group_id TEXT NOT NULL, node_id TEXT NOT NULL, role INTEGER DEFAULT 0, -- 0=member, 1=admin added_at INTEGER NOT NULL, PRIMARY KEY (group_id, node_id) ); CREATE TABLE peer_devices ( node_id TEXT PRIMARY KEY, display_name TEXT NOT NULL, identity_key BLOB NOT NULL, -- Ed25519 public key signed_pre_key BLOB, -- X3DH signed pre-key first_seen INTEGER NOT NULL, last_seen INTEGER NOT NULL, trust_level INTEGER DEFAULT 1 -- 0=blocked, 1=trusted, 2=verified );

9.2 Offline Queue

  • Messages sent while no peers are reachable are queued in the messages table with status = 0 (sending).
  • When a connection is established, the mesh layer scans for pending messages with matching conversation_id and sends them.
  • Queued messages are retained for 30 days, then purged.

9.3 Export / Backup

  • Full database export: encrypted SQLite dump, protected by user passphrase. Exported via local file sharing (USB, Bluetooth, or ChirpChat file transfer).
  • Individual conversation export: decrypted JSON format (plaintext). For user's own records.
  • Cloud backup: explicitly not supported (ISO_019). Backup is device-to-device.

10.3 Pairing Flow (User Perspective)

  1. User A taps "Pair with Device" on ChirpChat home screen.
  2. Phone emits a short ultrasonic chirp (inaudible). Screen shows "Listening for nearby devices..."
  3. User B taps "Pair with Device" on their phone. Phones are within ~5m.
  4. Both phones vibrate briefly to confirm pairing success. A new conversation appears in both users' conversation lists.
  5. If ultrasonic fails after 3 attempts, app auto-suggests QR code. User A shows QR, User B scans.
  6. Users can optionally verify keys by comparing a 6-digit code or a 12-word phrase shown on both devices.

10.4 Design Principles

  • Mobile-first: Primary UI targets phone screens (360-428dp width). Desktop is a secondary concern.
  • Offline-indicating: Connection state is always visible. Users should know when they're connected to the mesh and when they're isolated.
  • Battery-conscious: Ultrasonic chirping and mesh relay are power-intensive. UI should indicate battery impact and suggest disabling relay when not needed.
  • Privacy-respecting: No permission requests without explanation. Camera only for QR scanning. Microphone only during pairing. Location never required.

The following questions require experimentation and user testing to resolve:

  1. Ultrasonic reliability: How well do consumer phone speakers and microphones reproduce the 19-24kHz band? Measurement campaign needed across 50+ device models.
  2. Chirp range vs. noise: What is the effective range of ultrasonic pairing in quiet rooms vs. noisy environments (cafes, streets, wind)?
  3. WebRTC vs. raw TCP: Is WebRTC's DTLS+SCTP overhead acceptable for mesh routing, or should we use raw TCP with our own encryption?
  4. Mesh scaling: At what node count does flooding mesh break down? Simulation needed (100, 500, 1000 nodes).
  5. Erasure coding latency: What is the perceived delay for sending a 10MB photo through Reed-Solomon encode + mesh transmit + decode?
  6. iOS limitations: iOS does not expose Wi-Fi Direct APIs. What is the best alternative (Multipeer Connectivity, Bluetooth LE + mDNS over local Wi-Fi)?
  7. Battery drain: How much battery does mesh relay consume when the app is in the background? Does aggressive Doze mode on Android kill the relay service?
  8. Group key distribution: When removing a member, how quickly can the new group key be distributed to all remaining members over the mesh?
