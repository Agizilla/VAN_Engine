The Algorithm 3.8.2
Core: transition from CURRENT STATE to IDEAL STATE using verifiable criteria (ISC).
Goal: Euphoric Surprise — 9-10 ratings.

First Principles

Deletion over addition – before adding new code, consider deletion or refactoring. A fix that requires >5 new lines must first attempt to reduce existing complexity elsewhere.
Exact triggers only – all keywords that invoke special behavior must be all-caps, exact matches. No synonyms, no implied triggers. Example: GIT (not "commit"), DEPRECATE, ROLLBACK, RUN, PURGE.
PRD is truth – the PRD is the single source of truth for task quality and criteria. Sprint log tracks session lifecycle; PRD tracks task state.
Test failure returns to RISK – never jump back to BUILD directly. Re-evaluate assumptions first.

Every criterion must be ATOMIC — one verifiable end-state per criterion, 8-12 words, binary testable. See ISC Decomposition below.
Anti-criteria (ISC-A prefix): what must NOT happen.

ISC Decomposition Methodology
The core principle: each ISC criterion = one atomic verifiable thing. If a criterion can fail in two independent ways, it's two criteria. Granularity is not optional — it's what makes the system work. A PRD with 8 fat criteria is worse than one with 40 atomic criteria, because fat criteria hide unverified sub-requirements.
The Splitting Test — apply to EVERY criterion before finalizing

"And" / "With" test: If it contains "and", "with", "including", or "plus" joining two verifiable things → split into separate criteria.
Independent failure test: Can part A pass while part B fails? → they're separate criteria.
Scope word test: "All", "every", "complete", "full" → enumerate what "all" means. "All tests pass" for 4 test files = 4 criteria, one per file.
Domain boundary test: Does it cross UI/API/data/logic boundaries? → one criterion per boundary.

Context Compaction
The API does not expose current context usage as a measurable number, so use these heuristic triggers:

Phase-end compaction (mandatory for Extended+): After each phase transition, compact the prior phase's history into 1-3 paragraphs. Preserve: ISC status (pass/fail/pending), key results (numbers, decisions, code references), next actions. Discard: verbose tool output, intermediate reasoning, raw search results.
50% heuristic trigger: If accumulated conversation history visually exceeds ~50% of a typical session window (roughly: 50+ tool calls since last compaction, or assistant output is consistently truncated), pause and ask: "Context is accumulating — what should I preserve going forward? (a) Full ISC state + decisions, (b) Only pass/fail results, (c) Custom priority"
Post-approval compaction: Compact automatically if ALL criteria pass OR the user explicitly approves skipping further verification/testing.

This prevents context rot — degraded output quality from bloated history — which is the #1 cause of late-phase failures in long Algorithm runs.

Critical Rules (Zero Exceptions)

Exact trigger words — All trigger keywords are all-caps, exact match, no synonyms: GIT, ROLLBACK, DEPRECATE, RUN, PURGE. No implied triggers.
Mandatory output format — Every response MUST use exactly one of the output formats defined in the Execution Modes section of CLAUDE.md (ALGORITHM, NATIVE, ITERATION, or MINIMAL). No freeform output. No exceptions.
Response format before questions — Always complete the current response format output FIRST, then invoke AskUserQuestion at the end. Never interrupt or replace the response format to ask bare questions. Show your work-in-progress (OBSERVE output, reverse engineering, effort level, ISC, capability selection), THEN ask. Stopping the format to ask a bare question with no context is a failure.
Context compaction — Mandatory for Extended+. See Context Compaction section. This is the #1 prevention of late-phase failures in long Algorithm runs.
No Phantom Capabilities — Every capability listed under 🏹 CAPABILITIES SELECTED MUST be invoked via Skill tool call or Task tool call. Text output that resembles what the skill would produce is NOT invocation — it's theater. Selection without invocation is a CRITICAL FAILURE, checked at VERIFY.
Capability selection honesty — Do not select capabilities you don't intend to invoke. Zero selected is valid. Padding wastes time and consumes context.
No silent stalls — If an agent has been running for >30s without output, check its status. Ensure no processes are hung.
PRD is YOUR responsibility — If you don't edit the PRD, it doesn't get updated. There is no hook safety net. Every phase transition, every criterion check, every progress update — you do it with Edit/Write tools directly. If you skip it, the PRD stays stale. Period.
ISC Count Gate is mandatory — Cannot exit OBSERVE with fewer ISC than the effort tier floor (Standard: 8, Extended: 16, Advanced: 24, Deep: 40, Comprehensive: 64), unless the focused-task exception applies (min 4). If below floor, decompose until met. No exceptions.
Atomic criteria only — Every criterion must pass the Splitting Test. No compound criteria with "and"/"with" joining independent verifiables. No scope words ("all", "every") without enumeration.
Phase collapse by tier — Use the Phase Collapse by Tier table. Do not run phases collapsed for your tier. Show the correct phase count (e.g., ━━━ 3/10 ━━━ for Standard, not 3/21).
Before any code change: write to sprint_log.json — Status started. Update to done on completion. Never lose work.
Test failure → return to RISK, not BUILD — Re-evaluate assumptions before retrying implementation.
REFACTOR is mandatory for Extended+ — Scan for duplication >80% similarity. Skip only if user explicitly says "skip refactor".
